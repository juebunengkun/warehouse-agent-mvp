# 数仓建模规则

本项目的建模逻辑应围绕业务过程、事实表、维度表和稳定公共层展开，而不是直接把报表字段拼成一张宽表。

## 业务过程识别

- 订单、支付、退款、曝光、点击、访问、库存变更等属于业务过程。
- 业务过程决定事实表，而不是报表字段直接决定事实表。
- 一个复杂报表可能横跨多个业务过程，例如渠道经营日报同时涉及曝光、点击、下单、支付和退款。

## 事实表识别

- 事务事实表：一行一个事件，如订单、支付、曝光、点击。
- 周期快照事实表：一行一个周期状态，如账户余额、库存余额、用户当日状态。
- 累积快照事实表：一行描述流程生命周期，如下单到支付到发货到签收。
- 事实表粒度必须清楚，例如 `order_id`、`event_id`、`stat_date + sku_id`。

## 维度表识别

- 描述实体属性的是维度，如渠道名称、渠道类型、地区、省份、城市、用户标签、会员等级、商品类目。
- 低频变化维度适合 `_df` 全量快照。
- 需要历史还原的维度要考虑拉链表/SCD2。
- 维度缺失时使用 `unknown/default` 值兜底，而不是过滤事实数据。

## 增量/全量规则

- 事件事实表一般使用 `_di`，按业务日期增量产出。
- DWS 日汇总一般使用 `_di`。
- ADS 日报一般使用 `_di`。
- 地区、渠道、小规模字典维表一般使用 `_df`。
- 用户画像如果只取当天标签可以 `_df`；如果要还原历史标签要拉链。

## 依赖关系

- ODS 事实和 ODS 维度可以并行接入。
- DIM 要在 DWD/DWS 关联前产出。
- DWD 明细事实表可以先清洗事实字段，再关联 DIM 补充维度属性。
- DWS/ADS 依赖 DWD 和 DIM 产出完成。

## 表复用原则

表复用不能只看字段是否覆盖，必须检查：

- 字段覆盖
- 粒度一致或可安全上卷
- 指标口径一致
- 时间口径一致
- 业务过程一致
- 分区可用
- SLA 满足
- 是否认证公共表

字段不覆盖或粒度明显不一致时不能直接复用。未认证表可以降分，但需要暴露风险。

## Join 规则

- 事实表 join 维表默认 `LEFT JOIN`。
- Join key 必须来自事实表外键和维表主键。
- 避免一对多 join 导致指标膨胀。
- 维表 join 必须带分区条件，例如 `dim_channel_df.dt='${bizdate}'`。
- 维度缺失时使用默认值兜底，不应通过 `INNER JOIN` 过滤事实。
